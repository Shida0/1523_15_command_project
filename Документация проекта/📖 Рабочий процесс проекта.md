# 📖 Рабочий процесс проекта Asteroid Watch

> **Назначение документа:** Этот документ описывает как работает проект изнутри — от получения данных до отображения на сайте. Документ предназначен для команды разработчиков и поможет понять архитектуру и потоки данных.

---

## 🌟 Краткий обзор проекта

**Asteroid Watch** — это система мониторинга потенциально опасных астероидов (PHA). Мы собираем данные из трёх источников NASA, обрабатываем их и показываем пользователям через веб-интерфейс.

### Что делает система?

1. **Собирает данные** из NASA API (астероиды, сближения, угрозы)
2. **Обрабатывает и сохраняет** в базу данных PostgreSQL
3. **Предоставляет API** для фронтенда
4. **Обновляется автоматически** раз в 24 часа

### Какие данные обрабатываем?

| Тип данных | Источник | Что это | Сколько записей |
|------------|----------|---------|-----------------|
| **Астероиды** | NASA SBDB API | Орбиты, размеры, характеристики | ~2,500 PHA |
| **Сближения** | NASA CAD API | Когда астероиды пролетают мимо Земли | ~12,500 за 10 лет |
| **Угрозы** | NASA Sentry API | Вероятность столкновения | ~42 объекта с риском |

## 📁 Структура проекта

### Доменная архитектура

Проект разделён на **домены** по типу данных:

```
domains/
├── asteroid/     # Всё что связано с астероидами
├── approach/     # Всё что связано со сближениями
└── threat/       # Всё что связано с угрозами
```

Каждый домен содержит:
- **models/** — классы базы данных (таблицы)
- **schemas/** — схемы для валидации данных (Pydantic)
- **repositories/** — доступ к базе данных (CRUD операции)
- **services/** — бизнес-логика

### Общие компоненты (shared/)

```
shared/
├── config/           # Конфигурация приложения
├── database/         # Подключение к базе данных
├── external_api/     # Клиенты для NASA API
├── infrastructure/   # Базовые классы (сервисы, репозитории)
├── resilience/       # Circuit Breaker, Bulkhead, Timeout
├── transaction/      # Unit of Work для транзакций
└── utils/            # Вспомогательные функции
```

## 🔄 Полный цикл работы с данными

### Шаг 1: Получение списка астероидов из SBDB API

**Что происходит:**

1. Система подключается к NASA SBDB Query API
2. Запрашивает список всех PHA (Potentially Hazardous Asteroids) астероидов
3. Получает ~2,000-3,000 обозначений

**Технические детали:**

```
Запрос: GET https://ssd-api.jpl.nasa.gov/sbdb_query.api
Параметры:
  - fields: pdes (только обозначения)
  - sb-group: pha (только опасные астероиды)
  - limit: 3000

Ответ: ["433", "1862", "2023 DW", ...]
```

**Обработка:**

Для каждого астероида:
1. Отправляется детальный запрос к SBDB
2. Извлекаются параметры:
   - Орбитальные (перигелий, афелий, MOID)
   - Физические (диаметр, альбедо, магнитуда)
   - Класс орбиты (Apollo, Aten, Amor)
3. Если диаметр не указан — рассчитывается по формуле:
   ```
   diameter = 1329 / sqrt(albedo) * 10^(-0.2*H)
   ```

**Защита от блокировки:**

- Задержка 2 секунды между запросами
- Автоматические повторные попытки при ошибках
- Circuit Breaker для защиты от сбоев API

---

### Шаг 2: Получение данных о сближениях из CAD API

**Что происходит:**

1. Система подключается к NASA CAD (Close Approach Data) API
2. Запрашивает сближения на 10 лет вперёд
3. Фильтрует только для известных PHA астероидов

**Технические детали:**

```
Запрос: GET https://ssd-api.jpl.nasa.gov/cad.api
Параметры:
  - date-min: сегодня
  - date-max: +3650 дней (10 лет)
  - dist-max: 0.05 а.е. (только близкие)
  - body: Earth
  - sort: dist (сортировать по расстоянию)

Ответ: {
  "fields": ["des", "cd", "dist", "v_rel", "fullname"],
  "data": [
    ["2023 DW", "2023-Feb-14 19:46", 0.00078, 21.5, "2023 DW"],
    ...
  ]
}
```

**Обработка данных:**

1. Парсинг даты: `"2023-Feb-14 19:46"` → `datetime(2023, 2, 14, 19, 46)`
2. Конвертация расстояния: `0.00078 а.е.` → `117,000 км`
3. Группировка по астероидам
4. Фильтрация только для PHA из нашей базы

**Оптимизация:**

- Bulk-запросы для сохранения в базу
- Индексы на полях `asteroid_id` и `approach_time`
- Удаление старых сближений (которые уже прошли)

---

### Шаг 3: Получение оценок угроз из Sentry API

**Что происходит:**

1. Система подключается к NASA Sentry API
2. Запрашивает объекты с ненулевой вероятностью столкновения
3. Локализует данные (переводит на русский)

**Технические детали:**

```
Запрос: GET https://ssd-api.jpl.nasa.gov/sentry.api
Параметры:
  - ip-min: 1e-10 (минимальная вероятность)

Ответ: {
  "data": [{
    "des": "2023 DW",
    "ip": 0.002,        # Вероятность 0.2%
    "ts_max": 1,        # Туринская шкала
    "ps_max": -1.5,     # Палермская шкала
    "diameter": 0.05,   # Диаметр в км
    "energy_megatons": 50.0  # Энергия удара
  }]
}
```

**Локализация:**

Система переводит технические данные в понятный формат:

| Поле | Исходное | После обработки |
|------|----------|-----------------|
| **Туринская шкала** | `1` | "1 — Нормальный (зелёный)" |
| **Вероятность** | `0.002` | "0.2% (1 к 500)" |
| **Уровень угрозы** | — | "ОЧЕНЬ НИЗКИЙ" |

**Расчёт энергии:**

Если энергия не указана, рассчитывается по формуле:
```
energy = 0.5 * mass * velocity²
mass = volume * density
volume = 4/3 * π * (diameter/2)³
```

### Шаг 4: Сохранение в базу данных

**Архитектура сохранения:**

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  Астероиды   │────▶│  Сближения   │────▶│    Угрозы    │
│  (сначала)   │     │  (потом)     │     │  (в конце)   │
└──────────────┘     └──────────────┘     └──────────────┘
       │                    │                    │
       ▼                    ▼                    ▼
┌─────────────────────────────────────────────────────────────┐
│              Unit of Work (транзакция)                      │
│  • Все или ничего (атомарность)                             │
│  • Откат при ошибке                                         │
│  • Изоляция от других операций                              │
└─────────────────────────────────────────────────────────────┘
```

**Процесс сохранения:**

1. **Астероиды:**
   - Проверка на существование (по `designation`)
   - Если есть — обновление
   - Если нет — создание
   - Bulk-операция для скорости

2. **Сближения:**
   - Привязка к астероиду по `asteroid_id`
   - Проверка дубликатов (по дате и астероиду)
   - Удаление прошедших сближений

3. **Угрозы:**
   - Привязка к астероиду по `asteroid_id`
   - Обновление существующих оценок
   - Удаление угроз с истёкшими годами риска

**Оптимизация:**

- Индексы на всех полях поиска
- Connection pool для производительности
- Асинхронные операции для скорости

## 🔧 Механизмы отказоустойчивости

### Circuit Breaker

**Проблема:** Если NASA API не отвечает, система будет бесконечно пытаться подключиться.

**Решение:** Circuit Breaker автоматически прекращает запросы после нескольких неудач.

```
Состояния:
┌─────────┐    5 ошибок    ┌─────────┐   60 секунд   ┌──────────┐
│ CLOSED  │ ────────────▶ │   OPEN   │ ─────────────▶│ HALF_OPEN│
│(нормально)│              │(блокировка)│             │(проверка)│
└─────────┘                └─────────┘               └──────────┘
     ▲                                                  │
     │            успех                                 │ успех
     └──────────────────────────────────────────────────┘
```

**Настройки для NASA API:**
- Порог срабатывания: 3 ошибки
- Время восстановления: 60 секунд
- Типы ошибок: ConnectionError, TimeoutError

---

### Bulkhead

**Проблема:** Если один тип запросов "завис", он блокирует все остальные.

**Решение:** Bulkhead ограничивает количество одновременных запросов.

```
┌─────────────────────────────────────────┐
│           Bulkhead (max 5)              │
├─────────────────────────────────────────┤
│  [Запрос 1] [Запрос 2] [Запрос 3]       │
│  [Запрос 4] [Запрос 5]                  │
│  ────────────────────────────           │
│  [Запрос 6] ← в очереди (ждёт)          │
└─────────────────────────────────────────┘
```

**Настройки:**
- SBDB API: max 5 одновременных запросов
- CAD API: max 3 одновременных запроса
- Sentry API: max 2 одновременных запроса

---

### Timeout

**Проблема:** Запрос может "зависнуть" на неопределённое время.

**Решение:** Timeout автоматически прерывает запрос после указанного времени.

| API | Таймаут | Почему |
|-----|---------|--------|
| SBDB | 60 сек | Много данных |
| CAD | 120 сек | Сложные расчёты |
| Sentry | 180 сек | Очень сложный API |

## 📊 Обновление данных

### Автоматическое обновление

**Периодичность:** Раз в 24 часа

**Процесс:**

```
1. Запуск UpdateService
   │
   ├─▶ 2. Обновление астероидов
   │   ├─▶ Запрос к SBDB API
   │   ├─▶ Обработка ~2,500 астероидов
   │   └─▶ Сохранение в базу
   │
   ├─▶ 3. Обновление сближений
   │   ├─▶ Запрос к CAD API
   │   ├─▶ Обработка ~12,500 сближений
   │   └─▶ Удаление прошедших
   │
   └─▶ 4. Обновление угроз
       ├─▶ Запрос к Sentry API
       ├─▶ Обработка ~42 угроз
       └─▶ Удаление истёкших
```

**Логирование:**

Каждый шаг записывается в лог:
```
INFO: Начало обновления астероидов...
INFO: Получено 2,547 астероидов из NASA SBDB
INFO: Обновлено: создано 12, обновлено 2,535
INFO: Удалено 3 астероида, отсутствующих в NASA API
INFO: Обновление астероидов завершено
```

## 🎯 Связь между доменами

### Отношения моделей

```
┌─────────────────┐
│ AsteroidModel   │
│ (астероид)      │
└────────┬────────┘
         │
         │ 1 ко многим
         │
         ├──────────────────┐
         │                  │
         ▼                  ▼
┌─────────────────┐  ┌─────────────────┐
│CloseApproachModel│  │ThreatAssessment │
│   (сближения)   │  │    (угроза)     │
└─────────────────┘  └─────────────────┘
```

**Ограничения:**

- У астероида может быть **много** сближений
- У астероида может быть **одна** оценка угрозы
- При удалении астероида удаляются все связанные данные

### Транзакционность

Все операции с связанными данными выполняются в одной транзакции:

```python
async with UnitOfWork(session_factory) as uow:
    # 1. Создаём астероид
    asteroid = await uow.asteroid_repo.create(data)
    
    # 2. Создаём сближения (с ссылкой на астероид)
    for approach in approaches:
        approach['asteroid_id'] = asteroid.id
        await uow.approach_repo.create(approach)
    
    # 3. Всё или ничего
    await uow.commit()  # Если ошибка — откат всего
```

## 📈 Мониторинг и логирование

### Уровни логирования

| Уровень | Когда используется | Пример |
|---------|-------------------|--------|
| **DEBUG** | Отладка | "Запрос к API отправлен" |
| **INFO** | Информация | "Обновление завершено" |
| **WARNING** | Предупреждение | "Астероид без диаметра" |

Система логирует:
- Время выполнения запросов
- Количество обработанных записей
- Ошибки и повторные попытки

**Пример лога:**
```
INFO: GET /asteroids/all - 200 - 0.0234s
INFO: Обновление астероидов: 2547 записей за 45.2s
WARNING: Rate limit exceeded, waiting 65s
ERROR: Timeout error in SBDB API after 60s
```

## 🔐 Безопасность

### Защита данных

- **Пароли** хранятся в `config.yaml` (не в коде)
- **API ключи** не требуются (NASA API публичный)
- **Connection pool** ограничивает подключение к БД

### Rate Limiting

- Задержка между запросами к NASA: 2 секунды
- Максимум запросов в час: 1,000
- Автоматическая пауза при превышении лимита
